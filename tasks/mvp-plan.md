# MVP Implementation Plan — WC2026 Predictor

**Purpose:** Build a working end-to-end slice of the production system using 5 sample participants and 3 hand-authored matchday-1 results. Every artifact built here is reused unchanged in production. This document is self-contained — a competent developer with no prior context should be able to follow it start to finish.

**Time estimate:** ~9–10 hrs (one focused working day).
**Reference docs:** [docs/workflow.md](../docs/workflow.md) (system architecture), [asset/WC2026_Predictor_Game_Rules (v1).docx](../asset/) (scoring rules), [asset/WC2026_Predictor_Entry_Sheet (v1).xlsx](../asset/) (input template).

## 2. Parse template → fixtures.json (1 hr)

The Excel template is the canonical source of teams and group fixtures. Parse once, commit the JSON output, never duplicate the data.

### 2.1 Cell map (locked — see Appendix A)

Twelve group blocks at known cell ranges. Each block has 6 matches (rows × 2 score-input cells = 12 cells per block; 144 score cells total across the tournament).

### 2.2 Create `scripts/parse-template.js`

Required outputs:

**`data/fixtures/teams.json`**
```json
[
  { "code": "BRA", "name": "Brazil", "group": "C" },
  { "code": "USA", "name": "United States", "group": "D" },
  ...48 entries
]
```

Generate `code` as a 3-letter ISO-style abbreviation from the team name (use a hardcoded lookup table — keep it explicit, ~50 lines).

**`data/fixtures/groups.json`**
```json
[
  { "group": "A", "teams": ["MEX", "KOR", "ZAF", "CZE"] },
  ...12 entries
]
```

**`data/fixtures/matches.json`**
```json
[
  { "id": "G-A-1", "stage": "group", "group": "A", "home": "MEX", "away": "ZAF" },
  ...72 group matches followed by knockout placeholders:
  { "id": "R32-01", "stage": "r32", "label": "1A v Best 3rd", "home": null, "away": null },
  ...32 R32 + 16 R16 + 8 QF + 4 SF + 1 Final + 1 3rd-place = 104 total
]
```

### 2.3 Logic outline
1. Use SheetJS: `const wb = XLSX.readFile('asset/WC2026_Predictor_Entry_Sheet (v1).xlsx');`
2. Get first sheet: `const ws = wb.Sheets[wb.SheetNames[0]];`
3. For each of the 12 group blocks, read the home-team and away-team cells per row (e.g. for Group A row 5: `ws['A5'].v` and `ws['E5'].v`).
4. Build the team set by deduplicating across all group blocks → verify exactly 48 distinct names.
5. Assign group letter to each team based on which block it appeared in.
6. Map each team name to a 3-letter code via the lookup table.
7. Emit the three JSON files using `fs.writeFileSync` with 2-space indentation.

### 2.4 Validation (fail loudly if these fail)
- Exactly 48 unique teams
- Exactly 12 groups × 4 teams
- Exactly 72 group matches
- Every team in `groups.json` appears in `teams.json`
- Every team in match home/away appears in `teams.json`

### 2.5 Run
```
npm run parse-template
git add data/fixtures/
git commit -m "Generate fixtures from entry sheet template"
```

### 2.6 Acceptance
- Three JSON files exist in `data/fixtures/`
- Validation passes
- Files committed

## 3. Scoring engine + tests (2 hr) — **CRITICAL STEP**

This is the only piece that must be unambiguously correct. Build and test it *before* ingestion so you can validate ingested data against the scorer immediately.

### 3.1 Schemas

**Prediction JSON** (one per participant, lives at `data/predictions/<token>.json`):
```json
{
  "token": "8f3a2c9d1e4b7a6f2b9d8e7c1a3b4c5d",
  "handle": "varun_g",
  "name": "Varun Gupta",
  "submittedAt": "2026-06-08T14:23:00Z",
  "sourceFile": "Varun_Gupta.xlsx",
  "groupScores": {
    "G-A-1": { "home": 2, "away": 1 },
    "G-A-2": { "home": 0, "away": 0 },
    ...72 entries
  },
  "round32":      ["BRA", "ARG", "..." ],
  "round16":      ["BRA", "..."],
  "quarterFinal": ["BRA", "..."],
  "semiFinal":    ["BRA", "..."],
  "final":        ["BRA", "..."],
  "winner":       "BRA",
  "tiebreaker":   165,
  "warnings": []
}
```

**Result JSON** (one per match-day at `data/results/YYYY-MM-DD.json`):
```json
[
  {
    "matchId": "G-A-1",
    "stage": "group",
    "home": "MEX",
    "away": "ZAF",
    "homeScore": 2,
    "awayScore": 1,
    "goalsInExtraTime": 0,
    "shootoutGoals": 0,
    "winner": "MEX",
    "actualDate": "2026-06-11"
  }
]
```

For knockout matches, also include the `stage` (`r32`, `r16`, `qf`, `sf`, `final`, `3rd`) and `winner` (the team that progresses to the next round). The scorer only cares about `winner` for knockout stages.

### 3.2 Create `scripts/score.js`

**Public function:**
```js
function score(predictions, results, fixtures) -> Leaderboard
```

**`Leaderboard`** shape:
```js
[
  {
    token, handle,
    pointsByStage: { group: 47, r32: 25, r16: 30, qf: 15, sf: 0, final: 0, winner: 0 },
    totalPoints: 117,
    correctCounts: { groupResult: 14, groupExact: 5, r32: 5, r16: 3, qf: 1, sf: 0, final: 0 },
    tiebreaker: 165,
    tiebreakDelta: null  // populated only when ranking
  }
]
```

Sort by `totalPoints` desc, then by `tiebreakDelta` asc (if `actualTotalGoals` is known).

### 3.3 Scoring logic (encode verbatim from the rules doc)

**Group stage** — for each completed match in `results` where `stage === 'group'`:
- Look up the player's prediction at `groupScores[matchId]`
- If missing → 0 pts for that match
- If `prediction.home === actual.homeScore && prediction.away === actual.awayScore` → **5 pts** (exact score)
- Else if `predictedOutcome === actualOutcome` (W/L/D) → **3 pts** (correct result only)
- Else → 0 pts

Outcome helper:
```js
function outcome(h, a) { return h > a ? 'H' : h < a ? 'A' : 'D'; }
```

**Knockout stages** — for each stage in `['r32', 'r16', 'qf', 'sf', 'final']`:
- Let `actualTeams` = set of teams that **reached** this round (i.e. won the prior round, or for r32, qualified from groups). Extracted from `results`.
- Let `predictedTeams` = set from `prediction[stageKey]`.
- Points = `|predictedTeams ∩ actualTeams| × stagePoints`, where `stagePoints` = 5 / 10 / 15 / 20 / 25.

**Winner** — `+30` if `prediction.winner === actualWinner`.

**Tiebreaker** — store `prediction.tiebreaker` on the leaderboard entry. Computing `tiebreakDelta = |predicted - actualTotalGoals|` is the caller's responsibility (passed in optionally via a 4th arg `{ actualTotalGoals }`).

### 3.4 Create `scripts/score.test.js`

Use Node's built-in `node:test`:
```js
const { test } = require('node:test');
const assert = require('node:assert/strict');
const { score } = require('./score');
```

**Required test cases (must all pass):**

1. **Rules doc example 1:** Brazil 2-1 Morocco actual; prediction 3-1 Brazil → 3 pts (correct outcome, wrong score)
2. **Rules doc example 2:** Brazil 2-1 Morocco actual; prediction 2-1 Brazil → 5 pts (exact score)
3. **Rules doc example 3:** Brazil 2-1 Morocco actual; prediction 1-1 draw → 0 pts (wrong outcome)
4. **Draw both sides:** actual 0-0; prediction 0-0 → 5 pts
5. **Draw correct, score wrong:** actual 1-1; prediction 2-2 → 3 pts
6. **Missing prediction:** match in results but `groupScores[matchId]` undefined → 0 pts, no crash
7. **Knockout independence:** prediction has BRA in QF, actual results show BRA reached QF but lost; predicted opponent doesn't match → still 15 pts (R32 + R16 + QF if all predicted)
8. **Knockout subset:** if BRA predicted in QF but only reached R16 → 0 QF pts (but 10 R16 pts if BRA was also in `round16`)
9. **Tiebreaker:** two players on 100 pts, predicted goals 160 and 180, actual 170 → both have `tiebreakDelta` 10 → tied
10. **Winner correct:** prediction.winner = ARG, actual winner = ARG → +30
11. **Maximum score:** synthetic "perfect" prediction against synthetic "perfect" results → totalPoints = 960
12. **Empty results:** no completed matches → everyone has 0 pts
13. **Property: bounds:** for any prediction + any results, `0 ≤ totalPoints ≤ 960`

Run:
```
npm test
```

## 4. Ingestion: filled .xlsx → prediction JSON (1.5 hr)

`scripts/ingest.js` reads every `.xlsx` in `data/submissions/` and emits one `prediction.json` per file. Idempotent — running twice on the same file produces the same output.

### 4.1 Per-file algorithm

1. Open with SheetJS, take sheet 1.
2. Read `U2` → participant name. Empty → log error, skip file.
3. Compute token: `token = sha256(participantName + sourceFile).slice(0, 32)`. Deterministic = idempotent.
4. Derive handle: lowercase name, replace spaces/dots with `_`, strip non-alphanumeric (e.g. "Varun Gupta" → `varun_gupta`). If a collision occurs across two files, append `_2`, `_3`, etc.
5. Read group scores via the cell map (Appendix A). For each match, build the `matchId` deterministically from the group letter + row index.
6. Read knockout free-text cells (Appendix A):
   - R32: U8–U23 + X8–X23 (32 cells)
   - R16: U27–U34 + X27–X34 (16 cells)
   - QF: U38–U41 + X38–X41 (8 cells)
   - SF: U45–U46 + X45–X46 (4 cells)
   - Final: U50, U51 (2 cells)
   - Winner: X54 (1 cell)
7. For each knockout cell, normalize the free-text name (see 4.2).
8. Read `X57` → tiebreaker integer.
9. Run validation pass (see 4.3) — populate `warnings[]`.
10. Write `data/predictions/<token>.json`.

### 4.2 Team-name normalization

Pipeline applied to every free-text knockout entry:

1. **Trim** whitespace
2. **Strip diacritics**: `s.normalize('NFD').replace(/[̀-ͯ]/g, '')`
3. **Lowercase**
4. **Alias table lookup** — hardcoded map for known variants:
   ```js
   const aliases = {
     'usa': 'United States',
     'us': 'United States',
     'united states of america': 'United States',
     'south korea': 'South Korea',          // template uses "South Korea"
     'korea republic': 'South Korea',
     'czech republic': 'Czechia',
     'czechia': 'Czechia',
     'bosnia': 'Bosnia & Herz.',
     'bosnia and herzegovina': 'Bosnia & Herz.',
     'brasil': 'Brazil',
     'cote divoire': 'Ivory Coast',
     'cote d ivoire': 'Ivory Coast',
     ...
   };
   ```
5. **Exact match** against `fixtures/teams.json` names (case-insensitive after normalization)
6. **Levenshtein fallback** — distance ≤ 2 → accept; else → `null` + push warning `Unresolvable team name in <stage> cell <ref>: "<value>"`
7. **Output** the canonical team **code** (`BRA`, not `Brazil`) in the prediction JSON

### 4.3 Validation pass (warnings, not fatal)

After building the prediction object, check:
- All 72 group scores present and `0 ≤ score ≤ 30` (sanity bound)? Missing → warning.
- Exactly 32 R32 / 16 R16 / 8 QF / 4 SF / 2 Final / 1 Winner non-null entries? Otherwise → warning per missing slot.
- `round16` ⊆ `round32`, `quarterFinal` ⊆ `round16`, `semiFinal` ⊆ `quarterFinal`, `final` ⊆ `semiFinal`, `winner` ∈ `final`? Inconsistencies → warning per violation (player keeps the picks; they may still be scored against actuals).
- `tiebreaker` is a non-negative integer ≤ 500? Otherwise → warning.

### 4.4 Template-signature check

To detect a corrupted template:
```js
const A5 = ws['A5']?.v;
const expectedA5 = 'Mexico';  // from the v1 template
if (A5 !== expectedA5) {
  throw new Error(`Template signature mismatch in ${file}: A5=${A5}, expected ${expectedA5}. Did the participant edit team cells?`);
}
```

Anchor on multiple cells (e.g. also `G4 === 'GROUP B'`) for defence in depth.

### 4.5 Output log per file

Print one line per file:
```
[OK]   Varun_Gupta.xlsx       → varun_gupta (b3c1...)   72/72 group, 32/32/16/8/4/2/1 KO, 0 warnings
[WARN] Sarah_Lee.xlsx         → sarah_lee  (a2d4...)    71/72 group, 32/32/16/8/4/2/1 KO, 2 warnings
       - Missing group score: G-D-3
       - Fuzzy-matched "Brasil" → "Brazil" in QF cell U41
```

### 4.6 Create `scripts/ingest.test.js`

Tests:
1. **Round-trip:** ingest a known-good fixture file, assert all 72 group scores and all KO picks match expected
2. **Idempotency:** run ingestion twice; the two output JSONs are byte-identical
3. **Fuzzy match:** file with "Brasil" → resolves to "BRA", no warning (it's in the alias table) — or warning containing the word "Fuzzy" if matched via Levenshtein
4. **Unresolvable:** file with "Atlantis" → `null` in that slot + warning containing "Unresolvable"
5. **Missing name:** file with `U2` empty → throws error containing the filename
6. **Template signature mismatch:** edit `A5` to "TamperedTeam" → throws signature error
7. **Inconsistent bracket:** file where `quarterFinal` contains a team not in `round32` → warning containing "not in round32"

### 4.7 Acceptance
- All ingestion tests pass
- Running `npm run ingest` on a directory with 0 files exits cleanly with summary "0 files processed"

## Appendix B — Locked technical decisions

These are decided. Do **not** revisit during the build unless a fundamental blocker emerges; if you do revisit, capture the reason in [lessons.md](lessons.md).

1. **Single source of truth for fixtures** = the Excel template. Parsed once into `fixtures.json`; never hand-authored.
2. **Pure-function scoring.** `score()` has no I/O and no module-level state.
3. **Tokens** are sha256 of name + sourceFile, sliced to 32 hex chars. Deterministic → idempotent ingestion. Random-looking → not enumerable.
4. **Ingestion never blocks.** Errors become entries in `warnings[]` on the prediction. The only fatal cases are an empty name and a template signature mismatch.
5. **Free-text knockout normalization** is alias-table-first, Levenshtein-≤-2 second, `null` + warning third. Never guess.
6. **Predictions JSON is committed to the repo.** Source `.xlsx` files are committed for the MVP only.
7. **No client-side JavaScript on the site.** Everything statically rendered.
8. **All times UK (Europe/London).** Cron, digest send, displayed timestamps.
9. **Knockout points are per-team-in-round, independent of opponent.** A team predicted in QF scores 15 if it reaches QF via any path.
10. **Tailwind v3** (pinned to match pre-approved CLI invocation).
